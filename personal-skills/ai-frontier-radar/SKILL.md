# AI 前沿雷达 Skill

## 功能概述

针对默认重点跟踪厂商，在指定时间窗口内发现、核验、汇总确定性动态，标注来源类型与置信度，输出结构化 briefing。

**默认跟踪厂商以本文件当前列表与 source-map.md 中的独立章节为准；后续若新增厂商章节，需同步更新此列表，才视为纳入默认跟踪范围。当前为：**
OpenAI、Anthropic、Google/DeepMind、xAI、Meta AI、Perplexity、Mistral、月之暗面/Kimi、小米 Mimo、DeepSeek、智谱 AI、MiniMax

---

## 默认参数

| 参数 | 默认值 | 可覆盖方式 |
|------|--------|------------|
| 时间窗口 | 昨天 + 今天 | 用户直接说明，如"本周"、"2025-04-20 到今天" |
| 默认时区 | Asia/Shanghai | 用户明确指定时区则按用户要求 |
| 重点厂商 | 见上方列表 | 用户指定厂商名，则只关注指定厂商 |
| 输出语言 | 中文 | 用户说"英文"/"English"则切换 |
| 输出格式 | 结构化摘要 | 用户可要求"简版"/"纯列表"等 |

## 核心硬规则（最高优先级，不可绕过）

### 时间证据要求

**没有明确发布日期/时间证据的内容，不能进入"确定结论/重点事件/厂商动态"区。**

以下任一形式均可作为时间证据：
- 页面/文章上的明确发布日期
- GitHub Release/Commit 的时间戳
- Hugging Face 模型/数据集的更新日期
- 开放平台更新日志中的日期字段

不满足上述条件的内容：
- **只能放入"待核验线索"小节**，不得混入确定结论
- 或直接忽略，不在输出中提及

### 时间优先级规则

- **事件发生时间优先于报道时间**：以原始事件（发布、公告、上线）的时间作为"事件时间"
- 若无法获取原始事件时间，才退而采用媒体发布时间，并在来源处注明"以报道时间计"
- 避免"旧事新报"误判：若事件本身发生在时间窗口之外、只是今天被报道，不计入本期动态（除非明确说明"回顾"类需求）

### 时区规则

- 未指定时，以 **Asia/Shanghai (UTC+8)** 理解"今天/昨天"等相对时间
- 用户明确指定时区（如"按 UTC"）时，以用户要求为准
- 最终输出必须在时间窗口说明中注明所用时区

### 来源置信度规则

- `[官方]`：来自官方博客、官方公告、GitHub Release、Hugging Face 更新等厂商直接发布
- `[媒体报道]`：来自权威媒体（TechCrunch、The Verge、Reuters、Bloomberg、36kr 等）；即使是权威媒体，也只标 `[媒体报道]`，不能标 `[官方]`
- `[待核验]`：无明确日期证据、或来源为单一非权威渠道；**仅出现在"待核验线索"小节**，不得进入确定区
- 媒体报道可以进入"厂商动态"确定区，但须满足时间证据要求，且只标 `[媒体报道]`

## 执行流程

### 第一步：确定时间窗口

1. 读取用户输入，提取时间描述（"今天"、"昨天"、"本周"、具体日期）
2. 如未指定，默认使用：**昨天 00:00 ～ 今天当前时刻（Asia/Shanghai）**
3. 将时间窗口转为绝对日期范围（如 2025-04-28 ～ 2025-04-29）
4. 在后续搜索和筛选中始终带上此日期范围

### 第二步：信息检索（三层流程）

**第一层 — 候选发现（优先）：**

如果 tavily skill 可用：
```
加载 tavily skill，对每个厂商执行搜索：
- "{厂商名} news {日期范围}"
- "{厂商名} announcement {日期范围}"
- "{厂商名} 新闻 {日期}"（中文厂商附加）
```
使用 source-map.md 中各厂商的"优先搜索关键词"。

如果 tavily 不可用，**按以下顺序 fallback**（不要只依赖通用媒体聚合页）：
1. 对每个厂商，优先访问 source-map.md 中自动抓取适用性"高"或"中"的官方源
2. 若高/中适配度官方源未覆盖，再访问 source-map.md 中标注为"候选发现源"的官方入口页或组织页寻找线索
3. 若仍覆盖不足，再访问 source-map.md 中通用权威媒体源的 AI 栏目
4. 若某厂商在 source-map.md 中无高/中适配度官方源，在输出的来源汇总中注明"[无 tavily] 该厂商官方源覆盖质量有限"

**第二层 — 官方确认（必做）：**

对候选发现中的重要事件，回到官方源核验时间证据：
- 官方博客/公告页（见 source-map.md 中"自动抓取适用性：高/中"的源）
- GitHub Release 页（如 https://github.com/{org}/{repo}/releases）
- Hugging Face 模型页（如 https://huggingface.co/{org}）
- 开放平台更新日志（如有）

**第三层 — 补充扩展（可选）：**

如果前两层覆盖不足，可补充访问 source-map.md 中标注的其他媒体源。

> 注意：微信公众号等"手动核验源"不进入自动主流程，只作为人工参考。

### 第三步：处理 source-map 未覆盖厂商

当用户指定的厂商在 source-map.md 中无独立章节时：
1. **临时建立官方源候选**：通过搜索"{厂商名} official blog/news/docs/changelog"确认候选官方 URL
2. **执行相同确认流程**：访问候选 URL，核验时间证据
3. **若找不到稳定官方源**：在输出来源汇总处注明"[未收录厂商] {厂商名} 暂无稳定官方源，本期覆盖质量有限，结果以媒体报道为主"
4. 建议在后续请求中将该厂商补入 source-map.md

### 第四步：筛选与去重

1. **时间过滤**：只保留时间窗口内有明确日期证据的内容
2. **相关性过滤**：去除与 AI 无关的内容
3. **去重合并**：同一事件多个来源合并为一条
   - 保留一个"主来源"（最权威、最直接）
   - 必要时保留 1 条"佐证来源"
   - 不堆叠多条重复链接

### 第五步：验证来源

- 每条进入确定区的信息，必须有可访问的来源 URL 和明确时间证据
- 若只有媒体报道、无官方确认，标注 `[媒体报道]`，不标 `[官方]`
- 无法满足上述条件的，移入"待核验线索"或直接丢弃

### 第六步：处理无动态厂商

- 有确定动态的厂商：在"厂商动态"中单独列出
- 无确定动态的厂商：汇总到"本期暂无明确动态的厂商"小节，统一列名，不展开
- **不在每个厂商下单独写"本期暂无相关动态"**——避免输出被大量空条目淹没

## 输出结构

> 原则：**只列出本期有确定动态的厂商**，不预先写死所有厂商标题。

### 中文结构化摘要（默认）

```
## AI 前沿雷达 · {日期范围}（时区：Asia/Shanghai）

### 速览（TL;DR）
- {3-5 条最重要的确定事件，一句话概括}

---

### 重点事件

{按重要程度排列的重大事件，每条包含：事件描述、来源链接、置信度标签}

---

### 厂商动态

#### {厂商名 A}（仅列出本期有确定动态的厂商）
- {事件} — [来源名](URL) `[官方]`

#### {厂商名 B}
- {事件} — [来源名](URL) `[媒体报道]`

---

### 本期暂无明确动态的厂商

{无确定动态的厂商名称，逗号分隔}

---

### 为什么重要

{对本期最重要的 1-3 个事件，简要解释行业意义}

---

### 待核验线索

说明：以下内容未通过时间证据验证，仅供参考。
- {线索描述} — {来源} `[待核验]`

---

### 来源汇总

| 来源名 | 链接 | 来源类型 | 置信度 |
|--------|------|----------|--------|
| ... | ... | 官方博客/媒体报道/... | [官方]/[媒体报道] |
```

### 英文输出（用户要求"英文"/"English"时）

结构与中文版一致，所有标题、说明文字改为英文，厂商名使用英文别名（参考 source-map.md 各厂商别名字段）。

推荐标题映射：
- `速览（TL;DR）` → `Executive Summary`
- `重点事件` → `Key Events`
- `厂商动态` → `Company Updates`
- `本期暂无明确动态的厂商` → `Companies with No Clear Updates`
- `为什么重要` → `Why It Matters`
- `待核验线索` → `Leads Pending Verification`
- `来源汇总` → `Sources`

推荐标签映射：
- `[官方]` → `[Official]`
- `[媒体报道]` → `[Reported]`
- `[待核验]` → `[Needs Verification]`

### 简版（用户要求"简版"时）

保留标题、时间范围、时区说明；只输出"速览"和"厂商动态"两节，省略重点事件、为什么重要、待核验线索、来源汇总。

### 纯列表（用户要求"纯列表"时）

首行先写明时间范围与时区；每条动态格式：`{厂商名} · {事件一句话} · {来源链接} · {置信度标签}`，按厂商分组，无 Markdown 标题层级。

---

## 注意事项

- **不编造新闻**：没有找到时间证据的内容不得写入确定结论区
- **不过度总结**：保留关键细节（版本号、数字、时间等）
- **输出必须说明时间窗口和时区**
- 如搜索工具不可用，说明情况并提供手动查阅路径（参考 source-map.md）
- 输出长度：速览简洁，厂商动态适度展开，不堆砌
